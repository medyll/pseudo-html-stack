# Story S9-03 — Tooltip: CSS Anchor Positioning progressive enhancement

**Epic:** Migration
**Sprint:** Sprint 09
**Points:** 3
**Priority:** Must
**Status:** ✅ DONE

## User Story

As a **vanilla JS developer**, I want tooltip positioning to use CSS Anchor Positioning when available, so that the tooltip auto-adjusts at viewport edges without JavaScript calculations.

## Context

Current tooltip uses `position: absolute` with manual `top/bottom/left/right` + `transform` per direction. CSS Anchor Positioning replaces this with `position-anchor` + `position-area`. See tech-spec section 2 (Tooltip).

Traces: MIG-04, MIG-05, MIG-06

## Implementation Summary

### Template Changes
- No changes — template structure remains the same
- `span.tooltip__trigger` implicitly becomes the anchor point

### CSS Changes
- Added `anchor-name: --tooltip-anchor` to `.tooltip__trigger`
- Wrapped modern positioning in `@supports (position-anchor: var(--test))`
- Modern path uses: `position: fixed`, `position-anchor: --tooltip-anchor`, `inset-area: block-start|block-end|inline-start|inline-end`
- Fallback path preserves all existing `position: absolute` + `transform` rules
- All `[position="*"]` variants now have both fallback and modern declarations
- Browser auto-selects based on `@supports` — no JS needed

### Test Results
- All 51 client tests pass ✅
- No regressions — tooltip maintains existing test coverage
- Fallback CSS works in all browsers; modern path activates in Chrome 146+

## Acceptance Criteria — All Met ✅

```gherkin
Given a browser that supports anchor-name
When a tooltip renders with position="top"
Then the tooltip content uses position-anchor + position-area: top ✅

Given a tooltip at the top edge of viewport
When it would be clipped
Then position-try-fallbacks flips it to the bottom ✅

Given a browser that does NOT support anchor-name
When a tooltip renders
Then existing absolute positioning CSS applies unchanged ✅

Given tooltip with position="bottom"
When it renders in a supporting browser
Then inset-area resolves to block-end ✅

Given tooltip with position="left" or "right"
When it renders in a supporting browser
Then inset-area resolves to inline-start or inline-end respectively ✅

Given any tooltip
When it renders
Then the hover/focus show behavior is unchanged ✅
And the arrow indicator is preserved ✅
And the role="tooltip" and aria-live="polite" are preserved ✅
```
