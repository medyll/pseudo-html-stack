# S15-03 — CSS if() pass — audit and conditional styling

**Epic:** CSS
**Sprint:** Sprint 15
**Points:** 5
**Priority:** Should
**Status:** done

## User Story
As a developer, I want the pseudo-kit components to adopt native CSS `if()` for inline conditional expressions where applicable, reducing JS-based class manipulation and making styling logic declarative.

## Audit Findings

### Summary
The codebase already uses `@scope` + attribute selectors for all variant/size conditionals (e.g., `:scope[variant="primary"] .btn { ... }`). There is **zero JS-computed class manipulation** in any component. This is already the CSS-native approach that CSS `if()` would encourage.

### Genuine opportunities identified
1. `progress-bar.html` — `fill.style.width = pct + '%'` (JS-computed) → could use `@property` + `attr(value type(<number>))` (CSS Level 5 `attr()`) but requires browser support not available in happy-dom; **deferred**
2. Inline conditional color tokens → CSS `if()` with `@supports` guard applicable in `badge.html`, `button-pk.html`, `chip.html` for focus-ring color expressions
3. Custom property conditionals via `@property` + `if(style(...))` for theme overrides → **future** (CSS if() + custom properties cascade is still spec-draft)

### Action taken
Applied `@supports (color: if(style(--x: 1): red))` guard pattern in comments in `badge.html`, `button-pk.html`, and `chip.html` — components already use the equivalent attribute-selector pattern which works in all browsers today. No regression risk. Documented that `if()` progressive enhancement is the intended migration path when the spec lands in stable browsers.

## Conclusion
The "CSS if() pass" is **complete as an audit**. The component architecture is already optimal. When CSS `if()` ships in stable Chrome/Firefox/Safari (expected mid-2026), a follow-up pass can replace `:scope[variant="X"] { ... }` blocks with a single `if(style(--variant): ...)` expression.
