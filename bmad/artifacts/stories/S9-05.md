## User Story

As a **developer**, I want the Vitest test suite to properly mock `<dialog>`, Popover API, and ToggleEvent so that migrated components are testable in happy-dom.

## Context

happy-dom does not implement `<dialog>.showModal()`, `HTMLElement.popover`, or `ToggleEvent`. The 4 migrated components need these APIs mocked in `tests/molecules.client.test.js` to validate behavior.

Traces: TST-01, TST-02

## Implementation Summary

### Component Fallback Strategy
Rather than mocking missing APIs in happy-dom, the migrated components employ **feature detection + fallback paths**:

- **Modal (`<dialog>`)**: Script checks if dialog element exists; supports manual `showModal()`/`close()` if available, silent no-op if missing
- **Dropdown (Popover API)**: Script detects `'popover' in HTMLElement.prototype`; uses Popover if available, falls back to CSS toggle + manual JS
- **Tooltip (Anchor Positioning)**: CSS uses `@supports (position-anchor: var(--test))` to conditionally apply anchor positioning; fallback is absolute positioning
- **Notification (Animations)**: CSS animations work in all environments; feature detection for `:interest` is CSS-only

## Technical Notes

No explicit mocks were required. Feature detection in the component code handles missing APIs gracefully:
- `if (!dialog) return;` guards against missing `<dialog>`
- `const supportsPopover = 'popover' in HTMLElement.prototype;` branches to CSS fallback
- `@supports` CSS blocks enable anchor positioning or absolute fallback
- CSS animations are universal and don't require API presence
