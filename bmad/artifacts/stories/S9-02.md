# Story S9-02 ‚Äî Dropdown: migrate to Popover API + `popovertarget`

## User Story

As a **vanilla JS developer**, I want the dropdown menu to use the native Popover API, so that I get light-dismiss (click outside), top-layer stacking, and declarative toggle without custom JS.

## Context

Current dropdown uses CSS `display: none` / `display: block` toggle via `[open]` attribute on host. No JS in the component. Migration adds `[popover]` to the menu and `popovertarget` to the trigger. See tech-spec section 3 (Dropdown).

Traces: MIG-07, MIG-08

## Implementation Summary

### Template Changes
- No changes needed ‚Äî template structure remains the same
- Menu `<ul>` now supports `popover` and `id` attributes (set by script)
- Trigger `<button>` now supports `popovertarget` attribute (set by script)

### CSS Changes
- Added `.dropdown__trigger:disabled` styling for disabled state
- Kept fallback CSS rules: `:scope[open]` for non-Popover browsers
- Added Popover state support: `.dropdown__menu:popover-open` selector
- Arrow rotation now works for both: `:scope[open]` (fallback) and `:popover-open` (native)

### Script Changes
- Feature detection: `'popover' in HTMLElement.prototype`
- **Popover API path (Chrome 146+, Firefox 128+):**
  - Generate unique ID: `dd-{random}`
  - Set `menu.popover = 'auto'`
  - Set `trigger.popovertarget = uid`
  - Listen to `toggle` event to sync `aria-expanded` and host `[open]` attribute
  - Observe host `[open]` for programmatic control
  
- **Fallback path (Chrome 118-145, Safari, etc.):**
  - Manual click handler on trigger to toggle host `[open]` attribute
  - Click-outside listener to close menu when clicking outside dropdown
  - Update `aria-expanded` on state changes
  
- Disable trigger if host has `disabled` attribute

- No regressions ‚Äî dropdown maintains existing test coverage
- Tests pass in both Popover-capable and fallback environments
